## Problemas confirmados

### 1. Bloque 3D — Estrellas M activas (4 de 250 objetivo)

**Por qué falló:** El filtro `ew_espels_halpha < -1.0 AND ew_espels_halpha_flag = 0` es demasiado restrictivo. Las estrellas tipo M con emisión Hα intensa *y* medición de calidad perfecta son extremadamente raras en el catálogo. ESP-ELS tiende a asignar flags distintos de cero precisamente en las estrellas más activas, donde el ajuste del continuo es más difícil.

**Corrección sugerida en `gaia_queries.py`:**

python

```python
# ANTES
QUERY_BLOCK3D =f"""
SELECT TOP 250
{BASE_COLS_GS}
FROM gaiadr3.gaia_source AS gs
JOIN gaiadr3.astrophysical_parameters AS ap
    ON gs.source_id = ap.source_id
WHERE
{QUALITY_FLOOR_GS}
    AND gs.teff_gspphot < 4000
    AND ap.ew_espels_halpha IS NOT NULL
    AND ap.ew_espels_halpha < -1.0       -- demasiado restrictivo
    AND ap.ew_espels_halpha_flag = 0     -- demasiado restrictivo
ORDER BY gs.teff_gspphot ASC
"""

# DESPUÉS — umbral más suave y flags 0 y 1 aceptados
QUERY_BLOCK3D =f"""
SELECT TOP 250
{BASE_COLS_GS}
FROM gaiadr3.gaia_source AS gs
JOIN gaiadr3.astrophysical_parameters AS ap
    ON gs.source_id = ap.source_id
WHERE
{QUALITY_FLOOR_GS}
    AND gs.teff_gspphot < 4000
    AND ap.ew_espels_halpha IS NOT NULL
    AND ap.ew_espels_halpha < -0.5       -- umbral relajado
    AND ap.ew_espels_halpha_flag <= 1    -- acepta mediciones con advertencias menores
ORDER BY gs.teff_gspphot ASC
"""
```

**Impacto:** Recuperar ~200–240 objetos M activos. Si con este umbral sigue sin llegar a 250, es una limitación real del catálogo y habrá que documentarlo como caveat del dataset.

### 2. Bloque 5 — Ground truth GALAH (183 de 1.200 objetivo)

**Por qué falló:** El crossmatch DR2→DR3 se ejecutó durante el período de inestabilidad del servidor de ESA. Varios lotes probablemente fallaron silenciosamente o devolvieron resultados incompletos. El script `galah_crossmatch.py` necesita ser relanzado con el servidor estable.

**Corrección sugerida:** Relanzar `galah_crossmatch.py` completo cuando el archivo de ESA esté operativo (monitorear en [https://www.cosmos.esa.int/web/gaia/news](https://www.cosmos.esa.int/web/gaia/news)). El script ya está preparado — no requiere cambios de código.

bash

```bash
# Verificar que el servidor responde rápido antes de lanzar
# (si el smoke test tarda < 30s, el servidor está bien)
python gaia_queries.py --test

# Luego relanzar el crossmatch
python galah_crossmatch.py

# Y la descarga completa con el nuevo CSV
python gaia_queries.py --galah galah_gaia_dr3_ids.csv \
      --out gaia_sample_5000_final
```

**Objetivo mínimo:** 500 objetos con ground truth para poder calcular métricas confiables. Con 1.200 el benchmark es robusto.

### 3. Bloque 3C — Metal-pobres del halo (212 de 250 objetivo)

**Por qué falló:** Déficit menor (-38 objetos). El filtro `mh_gspphot < -1.0 AND phot_g_mean_mag < 16` es correcto físicamente pero la intersección es pequeña. No requiere cambio de lógica.

**Corrección sugerida:** Ampliar ligeramente el límite de magnitud:

python

```python
# ANTES
AND phot_g_mean_mag <16

# DESPUÉS
AND phot_g_mean_mag <17
```

Esto añade estrellas más débiles al pool, con la contrapartida de que tendrán menor cobertura de `fem_gspspec` (RVS). Dado que la cobertura de GSP-Spec ya es baja en general (1.3%), el impacto es marginal.

## Mejoras opcionales (no urgentes)

### 4. Cobertura de `fem_gspspec` y `alphafe_gspspec` (1.3%)

La cobertura es baja pero **esperada y correcta** — GSP-Spec requiere espectro RVS con G < ~16 y temperatura entre 3.500–8.000 K. No es un error del pipeline.

Para el sistema multi-agente esto significa que el `PhysicalContextAgent` operará principalmente con `mh_gspphot` (fotométrico) en lugar de `fem_gspspec` (espectroscópico) para la gran mayoría de las estrellas. El integrador deberá degradar la confianza en la metalicidad cuando solo tenga el proxy fotométrico, exactamente como el documento diseña.

Si se quiere aumentar la cobertura espectroscópica intencionalmente, se puede añadir un bloque específico de estrellas brillantes con RVS:

python

```python
QUERY_BLOCK_RVS_BRIGHT =f"""
SELECT TOP 300
{BASE_COLS}
FROM gaiadr3.gaia_source
WHERE
{QUALITY_FLOOR}
    AND phot_g_mean_mag < 13          -- rango óptimo del RVS
    AND teff_gspphot BETWEEN 3500 AND 8000
    AND ruwe < 1.4
ORDER BY phot_g_mean_mag ASC          -- las más brillantes primero
"""
```

Esto no está en la estratificación actual pero podría añadirse como "Bloque 6 — Estrellas RVS brillantes" si el `PhysicalContextAgent` necesita más ejemplos con `[Fe/H]` espectroscópico para desarrollarse.

### 5. Cobertura de tipos O/B (11 objetos, 0.3%)

Los tipos espectrales O y B están muy poco representados. Esto es físicamente correcto (son raros en el vecindario solar) pero limita la validación del caso Be del integrador. Si se quiere aumentar su representación:

python

```python
# Añadir sub-bloque específico para tipos calientes
QUERY_BLOCK_HOT =f"""
SELECT TOP 150
{BASE_COLS}
FROM gaiadr3.gaia_source
WHERE
{QUALITY_FLOOR}
    AND teff_gspphot > 10000
    AND logg_gspphot >= 3.5           -- evita supergigantes frías mal clasificadas
    AND ruwe < 1.4
    AND ebpminrp_gspphot < 0.5
ORDER BY source_id
"""
```

### 6. Verificar duplicados entre Bloque 5 y otros bloques

Una estrella GALAH puede coincidir con objetos ya descargados en los Bloques 1–4. El pipeline actual hace `drop_duplicates(subset='source_id')` que los elimina correctamente, pero vale la pena verificar cuántos objetos del Bloque 5 sobrevivieron la deduplicación para saber cuánto ground truth real queda.

python

```python
import pandas as pd
```
